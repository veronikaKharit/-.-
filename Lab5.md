# Лабораторная работа №5

## Тема
Реализация архитектуры на основе сервисов (микросервисной архитектуры)

## Цель работы
Получить опыт организации взаимодействия сервисов с использованием контейнеров Docker и настройки автоматической сборки и тестирования в GitHub Actions.

---

## 1. Краткое описание реализованной системы

В рамках лабораторной работы реализовано многоконтейнерное приложение для модуля диалогового тренажёра менеджеров по продажам. Предметная область и основные API-методы продолжают логику предыдущих лабораторных работ: в системе используются сценарии, тренировочные сессии, отправка ответа стажёра и завершение сессии. Такой набор методов уже был определён в лабораторной работе по REST API: `/api/v1/health`, `/api/v1/scenarios`, `/api/v1/scenarios/{scenarioId}`, `/api/v1/sessions`, `/api/v1/sessions/{sessionId}`, `/api/v1/sessions/{sessionId}/answer`, `/api/v1/sessions/{sessionId}/finish`.

---

## 2. Состав контейнеров

В проекте выделены следующие контейнеры:

1. **Frontend** — клиентская часть приложения. Отвечает за отображение интерфейса и отправку запросов в Backend API.
2. **Backend API** — единая точка входа в систему. Принимает HTTP-запросы, маршрутизирует их во внутренние сервисы и возвращает ответы клиенту.
3. **Training Service** — сервис, реализующий логику диалогового тренажёра: создание сценариев, запуск сессий, обработку ответа стажёра и завершение сессии.
4. **Chatbot Service** — отдельный сервис для поддержки сценария работы чат-бота.
5. **Database** — реляционная база данных, используемая для хранения сценариев и данных тренировочных сессий.

Минимальное требование лабораторной работы — наличие клиентской части, серверной части и базы данных — выполнено. Дополнительно система расширена двумя внутренними сервисами, что делает архитектуру ближе к реальной service-based модели.

---

## 3. Взаимодействие сервисов

Взаимодействие организовано следующим образом:

- пользователь работает с веб-интерфейсом;
- frontend отправляет запросы в Backend API;
- Backend API перенаправляет запросы в Training Service или Chatbot Service;
- Training Service и Chatbot Service при необходимости обращаются к базе данных;
- результаты возвращаются обратно через Backend API пользователю.

Таким образом, каждый контейнер отвечает за свою область ответственности. Это уменьшает связность системы и упрощает дальнейшее развитие проекта.

## 4. Запуск контейнеров

Для сборки и запуска приложения используется `docker-compose.yml`, расположенный в корне репозитория. В нём описаны все сервисы, их зависимости, сетевое взаимодействие и параметры запуска.

## 5. Непрерывная интеграция

В репозитории настроен workflow GitHub Actions, который автоматически запускается при `push` и `pull request`.

Основные шаги CI:

1. получение кода из репозитория;
2. настройка окружения Python;
3. настройка окружения Node.js;
4. установка зависимостей для тестирования;
5. установка Newman для запуска Postman-коллекции;
6. сборка и запуск контейнеров через Docker Compose;
7. ожидание готовности сервисов;
8. запуск интеграционных тестов `pytest`;
9. запуск Postman-тестов через Newman;
10. остановка контейнеров после завершения проверки.

Успешное завершение workflow в GitHub Actions подтверждает, что приложение корректно собирается, сервисы запускаются и основная функциональность работает.

## 6. Интеграционные тесты

### 6.1 Тесты на `pytest`

В проект включены интеграционные тесты на `pytest`, которые выполняются после запуска контейнеров. Они проверяют:

- доступность сервиса по endpoint `health`;
- основной сценарий работы тренировочного сервиса;
- один из рабочих сценариев взаимодействия с чат-ботом.

По результатам CI все `pytest`-тесты завершились успешно.

### 6.2 Postman-тесты

Дополнительно в проект добавлена коллекция Postman для проверки API. В репозитории она размещена в папке `postman/` и содержит:

- `lab5_postman_collection.json` — коллекция запросов;
- `lab5_postman_environment.json` — окружение с переменными.

В коллекции реализованы проверки следующих сценариев:

1. проверка состояния API (`GET /api/v1/health`);
2. создание сценария (`POST /api/v1/scenarios`);
3. получение сценария по идентификатору (`GET /api/v1/scenarios/{scenarioId}`);
4. создание сессии (`POST /api/v1/sessions`);
5. получение сессии по идентификатору (`GET /api/v1/sessions/{sessionId}`);
6. отправка ответа стажёра (`POST /api/v1/sessions/{sessionId}/answer`);
7. завершение сессии (`PUT /api/v1/sessions/{sessionId}/finish`).

Коллекция использует переменные окружения `scenarioId` и `sessionId`, которые автоматически сохраняются после выполнения предыдущих запросов и применяются в следующих шагах.

### 6.3 Запуск Postman-тестов в CI

Для автоматического выполнения Postman-коллекции в GitHub Actions подключён **Newman** — консольный runner для Postman.

Во время workflow после запуска контейнеров выполняется команда вида:

```bash
newman run postman/lab5_postman_collection.json -e postman/lab5_postman_environment.json
```

Это означает, что в процессе CI проверяются не только Python-интеграционные тесты, но и API-сценарии, оформленные в виде Postman-коллекции.

### 6.4 Пояснение по логам Newman

При выводе результатов Newman после каждого запроса показывает не только HTTP-ответ сервера, но и названия проверок (assertions). Например, запись вида:

```text
GET /api/v1/scenarios/ [405 Method Not Allowed]
1. Status code is 200
```

не означает, что сервер вернул код `200`. Она означает, что **проверка с названием `Status code is 200` завершилась ошибкой**, потому что фактический ответ был `405`. Реальное состояние теста определяется по статусу assertion и по итоговому блоку `failed / passed`.
